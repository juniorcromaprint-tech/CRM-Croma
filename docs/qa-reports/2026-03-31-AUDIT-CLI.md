## 3. Bugs Críticos (detalhe)

### BUG-01 — CampanhasPage: AlertDialogAction async sem preventDefault
**Arquivo**: `src/domains/comercial/pages/CampanhasPage.tsx`, linha 298-303

```tsx
<AlertDialogAction
  onClick={handleDisparar}   // ← handleDisparar é async (linha 134)
>
  Disparar
</AlertDialogAction>
```

`handleDisparar` é declarada como `async function handleDisparar()` na linha 134. O `AlertDialogAction` do Radix UI fecha o dialog imediatamente ao clicar, ANTES de qualquer `await` executar. Isso significa que o disparo de campanha pode falhar silenciosamente ou terminar em estado inconsistente.

**Correção**:
```tsx
<AlertDialogAction
  onClick={async (e) => {
    e.preventDefault();
    await handleDisparar();
    setConfirmOpen(false); // fechar manualmente
  }}
>
```

---

### BUG-02 — SharePropostaModal: update sem select().single()
**Arquivo**: `src/domains/comercial/components/SharePropostaModal.tsx`, linhas 32-36

```tsx
await supabase.from('propostas').update({
  share_token_active: true,
  share_token_expires_at: ...,
  status: 'enviada',
} as any).eq('id', propostaId);
// ← sem .select().single() — RLS silencioso
```

Se a policy RLS bloquear (ex: proposta de outro usuário), o update retorna 0 rows sem lançar erro. O modal continuaria como se o share tivesse sido ativado. Além disso, usa `as any` para contornar tipagem — indica problema de schema.

**Correção**: Adicionar `.select().single()` e tratar o erro retornado.

## 4. Bugs Médios (detalhe)

### BUG-03 — useAtividades: insert sem select().single()
**Arquivo**: `src/domains/comercial/hooks/useAtividades.ts`, linha 95

Atividades são registros de CRM (ligações, visitas, emails). Inserção sem `.select().single()` não detecta bloqueio RLS — atividade pode "sumir" sem feedback.

### BUG-04 — useCampanhas: dois updates sem select().single()
**Arquivo**: `src/domains/comercial/hooks/useCampanhas.ts`, linhas 64 e 79

- Linha 64: soft-delete de campanha (`.update({ excluido_em })`) — se falhar silenciosamente, usuário pensa que excluiu mas a campanha permanece ativa.
- Linha 79: atualização de campanha — dados de nome/configuração podem parecer salvo sem ter salvo.

### BUG-05 — AI Hooks migrados sem tratamento de erro próprio
**Arquivos**: `useAnalisarOrcamento.ts`, `useResumoCliente.ts`, `useBriefingProducao.ts`, `useComposicaoProduto.ts`, `useDetectarProblemas.ts`

Todos os 5 hooks foram migrados de `useMutation` direto para wrapper de `useAIBridge`. A migração removeu o `onError: (err) => showError(...)` que existia em cada um. O `useAIBridge` tem seu próprio tratamento de erro, mas sem a mensagem contextual específica de cada operação. Em caso de falha, o usuário verá um erro genérico (ou nenhum) em vez de "Erro ao analisar orçamento" / "Erro ao gerar briefing".

**Risco adicional**: como os hooks agora retornam `{ ...bridge, mutate, mutateAsync }`, o `isPending`, `isError` e `error` refletem o estado do `bridge` compartilhado — se dois hooks chamarem simultaneamente, os estados ficam misturados.

### BUG-06 — LeadDetailPage: segundo AlertDialogAction sem preventDefault
**Arquivo**: `src/domains/comercial/pages/LeadDetailPage.tsx`, linha 531-539

O primeiro `AlertDialogAction` da página (linha 499, conversão de lead) tem `e.preventDefault()` corretamente. O segundo (linha 531) não tem, chamando operação que pode ser async.

### BUG-07 — PedidosPage: AlertDialogAction exclusão sem preventDefault
**Arquivo**: `src/domains/pedidos/pages/PedidosPage.tsx`, linha 1119-1121

```tsx
<AlertDialogAction
  onClick={() => deleteId && deleteMutation.mutate(deleteId)}
>
```

Exclusão de pedido sem `e.preventDefault()`. Em conexão lenta, o dialog fecha imediatamente e o usuário não tem feedback da operação em andamento.
